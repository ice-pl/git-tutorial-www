Działania nasze
  1. aktualizowanie lokalnej wersji
    git pull
  2. tworzymy nową gałąź i na nią przechodzimy
    git branch feature
    git checkout feature
  3. tworyzmy nowy plik
    echo >> feature.txt
  4. dodajemy plik do kolejki
    git add .
  5. sprawdzamy bieżący status
    git status
  6. tworzymy commit
    git commit -m "Dodano nowy plik feature.txt"
  7. wgrywamy lokalne repozytorium na serwer (do odpowiedniej gałęzi)
    git push -u origin feature
    Jednorazowe polecenie, potem wystarczy tylko "git push".Opcja -u powoduje śledzenie gałęzi origin/master przez lokalną gałąź master.
    # master to nazwa gałęzi lokalnej.
  8. ...
git pull git branch feature git checkout feature git add . git status git commit -m "Dodano nowy plik feature.txt" git push -u origin feature

  2. inny użytkownik wchodzi do katalogu nowy_kolegi i przechodzi na gałąź feature
    cd nowy_kolegi
    git checkout feature
  3. inny użytkownik zmienia plik feature.txt
  4. ...

Działania nasze
  1. zmiana pliku feature.txt przez głównego użytkownika
  2. dodajemy plik do kolejki
    git add .
  3. sprawdzamy bieżący status
    git status
  4. tworzymy commit
    git commit -m "zmiana w drugiej linii"
  5. poprawiamy commit
    git commit --amend -m "zmiana w drugiej linii w pliku feature.txt"
    # gdy wykonaliśmy commit za wcześnie, a jeszcze potem doszły pewne drobne zmiany.
    Stara rewizja zostanie usunięta i powstanie nowa.
  6. wgrywamy lokalne repozytorium na serwer
    git push
  7. ...
git add . git status git commit -m "zmiana w drugiej linii" git commit --amend -m "zmiana w drugiej linii w pliku feature.txt" git push
Działania kolegi
  1. inny użytkownik dodaje plik do kolejki
    git add .
  2. inny użytkownik sprawdza bieżący status
    git status
  3. inny użytkownik tworzy commit
    git commit -m "zmiana w drugiej linii"
  4. inny użytkownik wgrywa lokalne repozytorium na serwer
    git push -u origin feature
    git wykrył, że wersja repozytorium zdalnego różni się od repozytorium lokalnego kolegi
    git informuje kolegę, że przyjmie zmiany gdy ten zaktualizuje swoją wersję projektu
  5. git nie wie, która wersja jest właściwa więc pokazuje obie:
    - w sekcji oznaczonej HEAD są zmiany wprowadzone przez kolegę
    - w sekcji oznaczonej hashem commitu są zmiany wprowadzone przez nas
  6. kolega decyduje, którą wersję pozostawić i usuwa niepotrzebny fragment i zatwierdza zmiany w reopzytorium
    git add .
    git status
    git commit -m "rozwiązono konflikt dotyczący drugiej linii pliku feature.txt"
    git push
git add . git status git commit -m "zmiana w drugiej linii" git push -u origin feature git commit -m "rozwiązono konflikt dotyczący drugiej linii pliku feature.txt" git push
Działania nasze
  1. pobieramy zmiany
    git pull
  2. przełączamy sie na gałąź master
    git checkout master
  3. połączenie brancha feature z masterem
    git merge feature
  4. usunięcie brancha feature
    git branch feature -d
  5. wgrywamy lokalne repozytorium na serwer
    git push
git pull git checkout master git merge feature git branch feature -d git push
